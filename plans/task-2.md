# План реализации Task 2: The Documentation Agent

## Цель

Превратить CLI из простого чат-бота в агента с инструментами для работы с файловой системой проекта.

## Архитектура

### 1. Схемы инструментов (Tool Schemas)

Определим два инструмента как функции Python и зарегистрируем их в LLM через function calling:

- **`read_file(path: str)`** — читает содержимое файла
  - Проверяет, что путь не выходит за пределы проекта (защита от `../`)
  - Возвращает содержимое файла или сообщение об ошибке

- **`list_files(path: str)`** —列出目录中的文件
  - Проверяет, что путь не выходит за пределы проекта
  - Возвращает список файлов/директорий через `\n`

### 2. Безопасность путей

Для защиты от выхода за пределы проекта:

- Используем `Path.resolve()` для получения абсолютного канонического пути
- Проверяем, что результат начинается с `PROJECT_ROOT`
- Блокируем пути с `..` компонентами

### 3. Агентовый цикл

```
1. Инициализация: messages = [{"role": "user", "content": question}]
2. Цикл (максимум 10 итераций):
   а. Отправить messages в LLM с определениями инструментов
   б. Если ответ содержит tool_calls:
      - Выполнить каждый инструмент
      - Добавить результат как {"role": "tool", ...}
      - Продолжить цикл
   в. Если ответ текстовый (message.content):
      - Извлечь answer и source
      - Вернуть JSON и выйти
3. Если достигнут лимит 10 вызовов — вернуть текущий ответ
```

### 4. Системный промпт

Инструктируем LLM:

- Использовать `list_files` для обнаружения файлов в `wiki/`
- Использовать `read_file` для чтения содержимого
- Включать ссылку на источник (файл + якорь раздела) в ответ

### 5. Формат вывода JSON

```json
{
  "answer": "...",
  "source": "wiki/git-workflow.md#resolving-merge-conflicts",
  "tool_calls": [
    {"tool": "list_files", "args": {"path": "wiki"}, "result": "..."},
    {"tool": "read_file", "args": {"path": "wiki/git-workflow.md"}, "result": "..."}
  ]
}
```
